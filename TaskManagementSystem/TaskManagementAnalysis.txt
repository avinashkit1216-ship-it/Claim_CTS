PROJECT OVERVIEW
================================================================================
The Task Management System is an enterprise-level learning project demonstrating
Clean Architecture principles in .NET. It showcases:

✓ 6-Layer Clean Architecture (Presentation, Application, Infrastructure, Domain)
✓ Clean Separation of Concerns (SoC)
✓ Dependency Injection (DI) - Inversion of Control (IoC)
✓ Repository Pattern for data abstraction
✓ Service Layer pattern for business logic
✓ Data Transfer Objects (DTOs) for API contracts
✓ Input validation with custom validators
✓ Unit and Integration testing with mocks
✓ REST API + MVC Web Application
✓ Entity Framework Core with In-Memory Database

Target: Demonstrate how to build scalable, testable, maintainable applications
Technology Stack: ASP.NET Core, C#, Entity Framework Core, xUnit, Moq

FILE: TaskManagement.Domain/Entities/TaskItem.cs
─────────────────────────────────────────────────
PURPOSE:
  Core business entity representing a task/todo item. This is the heart of the 
  domain layer and contains only business logic, with ZERO dependencies on 
  external frameworks.

  • Title (string): Required, task name

  • CreatedAt (DateTime): When task was created
  • UpdatedAt (DateTime?): When task was last modified

KEY METHODS:

PATTERNS USED:
  ✓ Entity Pattern: Pure business object
  ✓ Aggregate Root: TaskItem manages its own state
  ✓ Value Objects: Primitive types for fields

DEPENDENCIES:
  → None (0 external dependencies - independent & testable)

WHY SEPARATE:
  - Can use in console apps, mobile, or any .NET application
  - Business logic is protected from framework concerns
  - Easy to unit test without mocking
  - Changes in database strategy don't affect entity

CONNECTIONS:
  ← Used by: ITaskRepository, TaskService, Validators
  ← Used by: Tests (unit tests can instantiate directly)

FILE: TaskManagement.Application/DTOs/TaskDto.cs
─────────────────────────────────────────────────
PURPOSE:
  Data Transfer Objects separate API contracts from domain entities.
  Clients NEVER see internal entity structure.

DTO CLASSES:
  1. TaskDto (Read Model)
     - Id, Title, Description, IsCompleted
     - CreatedAt, UpdatedAt timestamps
     - Returned from GET operations

  2. CreateTaskDto (Write Model)
     - Only Title (required) and Description (optional)
     - Used in POST /api/tasks
     - Excludes Id, timestamps (server-generated)

  3. UpdateTaskDto (Write Model)
     - Title (required) and Description (optional)
     - Used in PUT /api/tasks/{id}
     - Excludes Id, timestamps

WHY USE DTOs:
  ✓ Security: Don't expose internal properties
  ✓ Flexibility: Change entity without breaking API
  ✓ Clarity: Define exactly what clients send/receive
  ✓ Validation: Input validation at API boundary
  ✓ Performance: Include only needed fields

PATTERN USED:
  ✓ DTO Pattern (Data Transfer Object)
  ✓ Separation of concerns: Entity ≠ API Contract

MAPPING:
  Infrastructure: Database (TaskItem) → Application (DTO) → Client (JSON)

CONNECTIONS:
  → Used by: Services, Controllers, Tests
  ← Mapped from: TaskItem entities

FILE: TaskManagement.Application/Services/TaskService.cs
──────────────────────────────────────────────────────────
PURPOSE:
  Core business logic orchestration. Coordinates between controllers, 
  validators, and repository. This is the "brain" of the application.

DEPENDENCIES INJECTED:
  • ITaskRepository: Data access abstraction
  • TaskValidator: Input validation

FLOW FOR EACH OPERATION:

  CreateTask Flow:
    1. Validate input using TaskValidator
    2. Throw ValidationException if invalid
    3. Create TaskItem entity
    4. Call repository.CreateAsync()
    5. Map entity to TaskDto
    6. Return TaskDto to controller

  UpdateTask Flow:
    1. Validate input
    2. Fetch existing task (throw NotFoundException if not found)
    3. Call task.Update() (entity's business logic)
    4. Call repository.UpdateAsync()
    5. Map to DTO and return

  DeleteTask Flow:
    1. Check if task exists (throw NotFoundException if not)
    2. Call repository.DeleteAsync()
    3. Return boolean success

  MarkAsCompletedTask Flow:
    1. Fetch task (throw NotFoundException if not found)
    2. Call task.MarkAsCompleted() (uses entity's method)
    3. Call repository.UpdateAsync()
    4. Return DTO

  GetAllTasks Flow:
    1. Fetch all from repository
    2. Map each entity to DTO
    3. Return collection

KEY CHARACTERISTICS:
  ✓ Depends on Domain + Shared (inward dependencies only)
  ✓ NO dependencies on Infrastructure or Presentation
  ✓ All public methods are async
  ✓ Delegates validation to TaskValidator
  ✓ Delegates data access to ITaskRepository
  ✓ Maps entities to DTOs before returning

PATTERN USED:
  ✓ Service Layer: Orchestrates business logic
  ✓ Dependency Injection: Receives collaborators via constructor
  ✓ Error Handling: Throws domain exceptions

WHAT MAKES THIS CLEAN:
  - Business logic is separate from HTTP concerns
  - Can test without database using mock repository
  - Can test without HTTP using dependency injection
  - High cohesion (one purpose: task operations)
  - Loose coupling (depends on interfaces, not implementations)

CONNECTIONS:
  → Depends on: ITaskRepository, TaskValidator, Domain.Entities
  ← Used by: Controllers (API and Web), Tests

FILE: TaskManagement.Infrastructure/Repositories/TaskRepository.cs
────────────────────────────────────────────────────────────────────
PURPOSE:
  Implements ITaskRepository interface. Encapsulates all database access 
  logic for tasks. This is the only place that knows about Entity Framework.

IMPLEMENTS: ITaskRepository

METHODS & IMPLEMENTATIONS:

  GetByIdAsync(int id):
    Query: _context.Tasks.FirstOrDefaultAsync(t => t.Id == id)
    Return: Single TaskItem or null
    Used by: Update, Delete operations

  GetAllAsync():
    Query: Fetch all tasks, ordered by CreatedAt descending
    Return: IEnumerable<TaskItem>
    Used by: List all tasks endpoint

  GetCompletedAsync():
    Query: WHERE IsCompleted = true, order by UpdatedAt descending
    Return: IEnumerable<TaskItem>
    Used by: Filter completed tasks endpoint

  CreateAsync(TaskItem task):
    Actions: 
      1. _context.Tasks.Add(task)
      2. await _context.SaveChangesAsync()
    Return: Created TaskItem with generated Id

  UpdateAsync(TaskItem task):
    Actions:
      1. _context.Tasks.Update(task)
      2. await _context.SaveChangesAsync()
    Return: Updated TaskItem
    Note: EF Core tracks changes automatically

  DeleteAsync(int id):
    Actions:
      1. Fetch task by id
      2. If exists: Remove and save
      3. If not: Return false
    Return: True if deleted, False if not found

  SaveChangesAsync():
    Explicit save method for manual change management
    Return: Number of rows affected

WHY SEPARATE FROM DOMAIN:
  ✓ Domain layer has NO database knowledge
  ✓ Can swap database implementation without changing business logic
  ✓ Infrastructure is testable with mock DbContext
  ✓ Isolates EF Core details

PATTERN USED:
  ✓ Repository Pattern: Encapsulates data access
  ✓ Async All The Way: Every operation is async
  ✓ Null Handling: Returns null instead of throwing
  ✓ Change Tracking: EF Core handles change detection

PERFORMANCE CONSIDERATIONS:
  ✓ Index on IsCompleted column for filtered queries efficiency
  ✓ Eager loading: GetByIdAsync is simple (no includes)
  ✓ Could add: Include() for related entities if needed

CONNECTIONS:
  → Implements: ITaskRepository (Domain)
  → Depends on: ApplicationDbContext
  ← Used by: TaskService, Tests (with mocks)

FILE: TaskManagement.API/Program.cs
────────────────────────────────────────
PURPOSE:
  Application startup configuration. Sets up Dependency Injection, 
  database, middleware, and routing.

KEY CONFIGURATIONS:

  1. SERVICE REGISTRATION (Dependency Injection):
     Services.AddControllers()              → Enable API controllers
     Services.AddDbContext()                → Register EF Core with in-memory DB
     Services.AddScoped<ITaskRepository, TaskRepository>  → Repository
     Services.AddScoped<TaskValidator>                    → Validator
     Services.AddScoped<ITaskService, TaskService>        → Service
     Services.AddEndpointsApiExplorer()     → API metadata
     Services.AddSwaggerGen()               → API documentation

  2. DATABASE INITIALIZATION:
     dbContext.Database.EnsureCreated()    → Create DB and seed data on startup

  3. MIDDLEWARE CONFIGURATION:
     UseSwagger()                           → Enable Swagger UI at /swagger
     UseSwaggerUI()                         → Interactive API documentation
     UseHttpsRedirection()                  → Force HTTPS
     UseAuthorization()                     → Auth checks
     MapControllers()                       → Route attribute-based controllers

HOW DEPENDENCY INJECTION WORKS:
  Example: When TasksController requests ITaskService:
    1. Controller constructor: public TasksController(ITaskService taskService)
    2. ASP.NET Core's DI container checks registrations
    3. Finds: services.AddScoped<ITaskService, TaskService>()
    4. TaskService needs ITaskRepository and TaskValidator
    5. Finds both in registrations
    6. Creates: TaskService(taskRepository, taskValidator)
    7. Injects into controller
    8. Request is handled

PATTERN USED:
  ✓ Composition Root: Program.cs is where dependencies are composed
  ✓ Dependency Injection: Loose coupling between components
  ✓ Service Locator: DI container resolves dependencies

STARTUP FLOW:
  Program.cs → Builds WebApplication → Configures middleware → Listens for requests

CONNECTION TO LAYERS:
  → Registers: TaskManagement.Application (Services, Validators)
  → Registers: TaskManagement.Infrastructure (Repository, DbContext)
  ← Controllers depend on registered services


FILE: TaskManagement.API/Controllers/TasksController.cs
───────────────────────────────────────────────────────
PURPOSE:
  Handles HTTP REST requests for task operations.
  Converts HTTP requests → service calls → HTTP responses.

REST ENDPOINTS:
  [ApiController]
  [Route("api/[controller]")]  → "api/tasks"

  ENDPOINTS DEFINED:

  1. GET /api/tasks
     Method: GetAllTasks()
     Returns: 200 OK with list of TaskDto
     Handles: Fetch all tasks

  2. GET /api/tasks/{id}
     Method: GetTaskById(int id)
     Returns: 200 OK + TaskDto OR 404 Not Found
     Handles: Fetch single task

  3. POST /api/tasks
     Method: CreateTask(CreateTaskDto)
     Returns: 201 Created + TaskDto + Location header
     Body: { "title": "...", "description": "..." }
     Handles: Create new task

  4. PUT /api/tasks/{id}
     Returns: 200 OK + updated TaskDto OR 400/404
     Handles: Update existing task

  5. DELETE /api/tasks/{id}
     Returns: 204 No Content OR 404 Not Found
     Handles: Delete task

  6. PUT /api/tasks/{id}/complete
     Returns: 200 OK + TaskDto
     Handles: Mark task as completed

DEPENDENCY INJECTION:
  Constructor: public TasksController(ITaskService taskService)
  DI Container provides implementation automatically
  Loose coupling: Controller depends on interface, not concrete class

ERROR HANDLING:
  try-catch blocks catch exceptions:
    • ValidationException → 400 Bad Request
    • NotFoundException → 404 Not Found
    • Other exceptions → 500 Internal Server Error

RESPONSE TYPE DOCUMENTATION:
  [ProducesResponseType(typeof(TaskDto), StatusCodes.Status200OK)]
  [ProducesResponseType(StatusCodes.Status404NotFound)]
  These attributes document API for Swagger

PATTERN USED:
  ✓ REST API Pattern: HTTP methods map to CRUD operations
  ✓ Dependency Injection: Service injected via constructor
  ✓ Exception Handling: Converts domain exceptions to HTTP responses
  ✓ API Documentation: ProducesResponseType attributes

HTTP STATUS CODES:
  200 OK              → Operation successful
  201 Created         → Resource created, returns Location header
  204 No Content      → Operation successful, no body to return
  400 Bad Request     → Validation error
  404 Not Found       → Resource doesn't exist
  500 Server Error    → Unexpected error

KEY RESPONSIBILITY:
  ✓ Accept HTTP requests
  ✓ Validate basic input (ModelState)
  ✓ Delegate to service
  ✓ Catch domain exceptions
  ✓ Return appropriate HTTP responses
  ✗ NOT: Handle business logic (that's service layer)
  ✗ NOT: Access database (that's repository layer)

CONNECTIONS:
  → Depends on: ITaskService (Application layer)
  ← Called by: HTTP clients (browsers, mobile apps, other services)

=================================================================================
ARCHITECTURAL PATTERNS USED
=================================================================================

1. CLEAN ARCHITECTURE (Layered Architecture)
   ────────────────────────────────────────────
   Principle: Dependencies point INWARD
   Benefit: Each layer ignorant of outer layers
   
   Dependency Diagram:
     Controllers     (Web/API)
        ↑ ↓
     Services        (Application)
        ↑ ↓
     Repositories    (Infrastructure)
        ↑ ↓
     Entities        (Domain)

2. REPOSITORY PATTERN
   ─────────────────
   What: ITaskRepository abstracts data access
   Why: Swap database without changing business logic
   How: Service depends on ITaskRepository interface
   
   Benefit: Can mock repository for testing

3. Service Layer PATTERN
   ────────────────────
   What: TaskService orchestrates business operations
   Why: Isolate business logic from controllers
   How: Controllers delegate to services
   
   Benefit: Reusable logic for both API and Web

4. DEPENDENCY INJECTION (Inversion of Control)
   ──────────────────────────────────────────
   What: Components receive dependencies, don't create them
   How: ASP.NET Core DI container resolves dependencies
   Why: Loose coupling, easy testing with mocks
   
   In Program.cs:
     AddScoped<ITaskService, TaskService>()
     AddScoped<ITaskRepository, TaskRepository>()

5. DTO (Data Transfer Object) PATTERN
   ────────────────────────────────────
   What: TaskDto represents API contract
   Why: Protect domain entity from external exposure
   How: Map Entity → DTO before returning from API
   
   Benefit: Controllers never expose internal entity structure

6. MAPPER PATTERN
   ──────────────
   What: MapToDto(TaskItem) converts entity to DTO
   Why: Separate domain model from API representation
   How: Private static method in service
   
   Benefit: Single source for Entity → DTO mapping logic

7. VALIDATOR PATTERN
   ──────────────────
   What: TaskValidator encapsulates validation rules
   Why: Reusable validation, testable in isolation
   How: Service calls validator before creating entity
   
   Benefit: Easy to modify validation rules

8. EXCEPTION-BASED ERROR HANDLING
   ────────────────────────────────
   What: Custom exceptions (NotFoundException, ValidationException)
   Why: Controllers can catch specific exception types
   How: throw new ValidationException() in service
   Provider: Mapping to HTTP status codes in controller
   
   ValidationException → 400 Bad Request
   NotFoundException → 404 Not Found

9. ASYNC/AWAIT PATTERN
   ────────────────────
   What: All I/O operations are asynchronous
   Why: Non-blocking, better performance
   How: async/await keywords throughout
   
   Benefits:
     ✓ Scalability: handle more concurrent requests
     ✓ Resource efficiency: threads not blocked on I/O

10. COMMAND/QUERY SEPARATION (CQRS-lite)
    ───────────────────────────────────
    Command Operations (modify data):
      CreateTaskAsync(), UpdateTaskAsync(), DeleteTaskAsync()
    
    Query Operations (read data):
      GetAllTasksAsync(), GetTaskByIdAsync()
    
    Benefit: Clear distinction of intent

BEST PRACTICES DEMONSTRATED
=================================================================================

✓ SEPARATION OF CONCERNS
  Each layer has one responsibility:
  • Domain: Business rules
  • Application: Orchestration & validation
  • Infrastructure: Data access
  • Presentation: HTTP handling
  Result: Easy to modify one layer without affecting others

✓ DEPENDENCY INVERSION
  Depend on abstractions (interfaces), not concretions
  Example: Service depends on ITaskRepository, not TaskRepository
  Result: Easy to swap implementations, testable

✓ DRY (Don't Repeat Yourself)
  Constants defined once: AppConstants.cs
  Validation logic centralized: ValidationHelper, TaskValidator
  Mapping logic in one place: MapToDto()
  Result: Changes propagate everywhere automatically

✓ SINGLE RESPONSIBILITY PRINCIPLE
  Each class has one reason to change:
  • TaskService: task operations logic
  • TaskValidator: task validation rules
  • TaskRepository: data access
  • TasksController: HTTP handling
  Result: Easy to test, understand, and maintain

✓ OPEN/CLOSED PRINCIPLE
  Open for extension, closed for modification
  Example: Add new entity type
    • Create Entity, Enum, Interface
    • Create Service, Validator, Repository
    • Create Controller
    • Existing code unchanged
  Result: Minimal risk when extending

✓ ASYNC ALL THE WAY
  All I/O operations are asynchronous
  Repository methods: async Task<T>
  Service methods: async Task<T>
  Controller actions: async Task<IActionResult>
  Result: Better scalability and resource utilization

✓ ERROR HANDLING
  Custom exceptions are specific:
  • ValidationException: validation failures → 400
  • NotFoundException: resource missing → 404
  Controllers map exceptions to HTTP statuses
  Result: Meaningful error responses

✓ VALIDATION LAYERING
  Input validated at API boundary (ModelState)
  Business rules validated in service (validators)
  Entity enforces constraints (constructors, methods)
  Result: Multiple checkpoints for data integrity

✓ TESTABILITY
  Unit tests mock dependencies
  Integration tests use real components (in-memory DB)
  Test coverage for:
    • Business logic (service tests)
    • Data access (repository with mock DbContext)
    • HTTP endpoints (controller integration tests)
  Result: High confidence in code quality

✓ DOCUMENTATION
  XML comments on every class/method
  Example:
    /// <summary>
    /// Create a new task
    /// </summary>
    public async Task<TaskDto> CreateTaskAsync(...)
  Swagger generates API documentation automatically
  Result: Self-documenting code

POTENTIAL IMPROVEMENTS & RECOMMENDATIONS

1. ✓ IMPLEMENT PAGING FOR GetAll() METHODS

   Risk: Performance issues with large datasets

   Example:
     RuleFor(x => x.Title)
       .NotEmpty()
       .Length(3, 200);
   
   Benefit: More powerful, consistent validation library

   Current: Manual MapToDto() method
   Better: AutoMapper NuGet package
   
   Configuration:
     CreateMap<TaskItem, TaskDto>();
   
   Benefit: Less boilerplate, easier to maintain

   Current: TaskController uses DTOs directly
   Better: Create separate ViewModels
   
   Example:
     TaskListViewModel: IEnumerable<TaskDto>

   Current: No logging
   Better: Serilog or Microsoft.Extensions.Logging

     _logger.LogInformation("Task created: {TaskId}", taskItem.Id);

   Benefit: Troubleshoot issues, monitor application

   Current: Hard delete removes data
   Better: Add IsDeleted flag, only logical delete
   
   Schema:
     IsDeleted: bool = false

   Benefit: Data recovery, audit trail

7. ✓ USE MEDIATOR PATTERN (MediatR)
   Current: Services directly injected
   Better: CQRS with MediatR library

     public class CreateTaskCommand : IRequest<TaskDto>

     public class CreateTaskCommandHandler

     var result = await _mediator.Send(new CreateTaskCommand(...));
   
   Benefit: Better organize complex logic, easier testing

8. ✓ IMPLEMENT CACHING
   Current: Every request hits database
   Better: Cache GetAllTasks() result
   
   Implementation:
     [OutputCache(Duration = 60)]
     public async Task<IActionResult> GetAllTasks()
   
   Benefit: Better performance, reduced database load

9. ✓ IMPLEMENT AUTHENTICATION/AUTHORIZATION
   Current: No security
   Better: Add JWT authentication, role-based authorization
   
   Attributes:
     [Authorize]

10. ✓ USE SPECIFICATION PATTERN FOR COMPLEX QUERIES

    Current: No versioning

    Routes:
      /api/v1/tasks

    Current: try-catch in each controller
    Better: Custom middleware layer
    
    Middleware:
      app.UseExceptionHandler() + custom handler

    Current: Standalone application

SCALABILITY & PERFORMANCE CONSIDERATIONS
=================================================================================

CURRENT ARCHITECTURE SCALES BY:

1. STATELESS DESIGN
   ✓ No server-side state in controllers
   ✓ Can run multiple instances behind load balancer
   ✓ Easy horizontal scaling

2. DEPENDENCY INJECTION SCOPING
   ✓ Scoped services created per request
   ✓ Efficient resource management
   ✓ DbContext lifecycle tied to request

3. ASYNC/AWAIT THROUGHOUT
   ✓ Non-blocking I/O
   ✓ Thread pool threads available for other requests
   ✓ Better thread utilization

4. ENTITY FRAMEWORK QUERY OPTIMIZATION
   ✓ Index on IsCompleted column
   ✓ No unnecessary joins (lazy loading disabled by default)
   ✓ Can add eager loading with Include() when needed

RECOMMENDATIONS FOR SCALE:

1. Add caching layer (Redis)
   • Cache GetAllTasks() results
   • Invalidate on create/update/delete
   • Significant performance boost

2. Implement pagination
   • Limit result sets
   • GetAllTasks(pageNumber=1, pageSize=10)
   • Reduce bandwidth and memory usage

3. Use Read/Write database splitting
   • Read-only replicas for queries
   • Write database for inserts/updates
   • Better performance under high load

4. Implement event sourcing
   • Store state changes as events
   • Rebuild state from events when needed
   • Audit trail built-in

5. Add background job processing
   • Long-running operations in background
   • Return 202 Accepted immediately
   • Process asynchronously with Hangfire/Quartz
